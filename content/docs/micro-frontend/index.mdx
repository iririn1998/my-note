# マイクロフロントエンドとは

**マイクロフロントエンド（Microfrontends）**とは、大規模なフロントエンドアプリケーションを**小さな独立した部分に分割し、それぞれを個別に開発・テスト・デプロイできる**ようにする設計原則およびアーキテクチャのことです。

バックエンドにおける「マイクロサービス」の概念をフロントエンドにも適用したもので、複数のチームが異なる技術スタックを用いて、一つのまとまったアプリケーションを構築することを可能にします。

## 基本的な考え方と3原則

マイクロフロントエンドは、アプリケーションを「境界づけられたコンテキスト」というビジネス機能単位で分割します。一般的に以下の3つの原則が重要視されています。

*   **技術スタックの自由:** 各チームは、他のチームに影響を与えることなく、React、Vue.js、Angularなど、自分たちのニーズに最適なフレームワークやライブラリを自由に選択できます。
*   **独立したコードとルール:** 各コンポーネントは独自のコードベースを持ち、状態やグローバル変数を共有せず、独立して機能するように管理されます。
*   **独立したデプロイ:** 全体をビルドし直すことなく、特定の機能（マイクロフロントエンド）だけを更新・デプロイできます。

## なぜマイクロフロントエンドが必要なのか（メリット）

従来の「モノリス（一枚岩）」なフロントエンド開発が抱える課題を解決するために採用されます。

*   **開発速度と生産性の向上:** チーム間の依存関係が減るため、並行して開発を進められ、ビルドやデプロイの時間も短縮されます。
*   **スケーラビリティ:** 大規模な組織において、特定の機能（例：検索機能、ショッピングカートなど）ごとに担当チームを分けてスケールさせることが容易になります。
*   **段階的な移行（Incremental Migration）:** 古いレガシーシステムを一度に書き換えるのではなく、一部の機能から徐々に新しい技術へ移行することができます。
*   **リスクの軽減:** 特定のマイクロフロントエンドで問題が発生しても、アプリ全体を停止させることなく、その部分だけを修正・アップデートできます。

## 技術的な実現方法

大きく分けて、サービスをどう分けるか（分割）と、どう組み合わせるか（統合）の視点があります。

### 分割方法

*   **垂直分割:** 「商品ページ」「決済ページ」のようにURL（パス）ごとにアプリケーションを分ける方法です。
*   **水平分割:** 1つのページの中に、ヘッダー、物件検索、お気に入りリストなど、複数の小さな断片（フラグメント）を配置する方法です。

### 統合方法

*   **クライアント側統合:** Web Components（Custom Elements）やJavaScriptを使用してブラウザ上で統合します。Vercelの**Remote Components**もこの一種で、外部URLからコンポーネントを動的に取得します。
*   **サーバー側統合:** SSR（サーバーサイドレンダリング）やSSI（Server Side Includes）を用いて、サーバーがページを配信する際に統合します。
*   **iframe:** 各機能をiframeで呼び出す、最も古典的で分離レベルの高い方法です。

## 課題と注意点（デメリット）

メリットがある一方で、以下のようなトレードオフも存在します。

*   **複雑性の増加:** 複数のアプリを統合・管理するためのインフラや共通ルールの整備が必要になり、アーキテクチャ全体が複雑になります。
*   **パフォーマンスへの影響:** 各アプリが独自のライブラリを読み込むと、全体のファイルサイズ（ペイロード）が肥大化し、読み込みが遅くなる可能性があります。
*   **運用・管理コスト:** チームごとにデプロイフローや監視体制を整える必要があり、ガバナンスの維持が難しくなります。

## 比喩による解説

マイクロフロントエンドは、**「大型モジュール式のショッピングモール」**に例えられます。

モール全体（一つのWebサイト）の中に、「衣料品店」「飲食店」「映画館」といった専門店（マイクロフロントエンド）が独立して入っているような状態です。それぞれの店は独自の内装（技術スタック）や店員（チーム）を持ち、他の店を閉めることなく個別に改装（アップデート）できます。客（ユーザー）からは、一つの大きな施設としてシームレスに利用できるように統合されています。
